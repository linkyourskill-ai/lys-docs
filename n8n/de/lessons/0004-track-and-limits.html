Dein Agent sucht und beauftragt. Was noch fehlt: der Überblick danach. Heute lässt du ihn seine Aufträge selbst verfolgen, lernst die englischen Status-Strings der Plattform zu lesen — und die Grenzen kennen, in denen dein Agent arbeitet. Am Ende ist dein Kurs-Agent komplett.

Lektion 04 · ca. 8 Minuten · Wissen & Übung

Das kannst du danach

  • Aufträge per Chat verfolgen — mit get_order_status und list_orders.
  • Die Status-Strings der Plattform lesen und den deutschen Badges zuordnen.
  • Die zwei Grenz-Arten unterscheiden: deine Auftrags-Limits und die Rate-Limits der Plattform.

Die letzten zwei Haken

  1. Öffne den Node MCP Client Tool und ergänze unter Tools to Include → Selected die Tools get_order_status und list_orders.
  2. Das ist die End-Ausstattung deines Kurs-Agenten — fünf Tools: suchen, Bewertungen lesen, einen Auftrag vorbereiten, Status abfragen, Aufträge auflisten.

Kerngedanke

Fünf Tools reichen — und die Auswahl ist bewusst: Alles, was an deiner Freigabe vorbeiführen könnte (etwa die task_*-Tools des Katalogs), bleibt aus. Dein Agent kann damit alles vorbereiten und alles nachlesen, aber nichts entscheiden. Entscheiden bleibt dein Tool.

Dein Agent verfolgt seine Aufträge

Frag im Chat: Zeig mir meine letzten Aufträge. Dein Agent ruft list_orders auf und liefert die Liste — darin solltest du den Übungsauftrag aus Lektion 3 wiederfinden. Dann frag nach: Wie ist der Status von „Test: Fensterreinigung"? Jetzt antwortet get_order_status.

Dabei fällt dir etwas auf: Die Plattform antwortet auf Englisch. Auf der technischen Leitung heißen die Stationen des Agenten-Wegs:

  • prepared — der Auftrag ist vorbereitet; im Web kennst du ihn als Badge „Vorbereitet".
  • approval_requested — die Plattform hat deine Freigabe angefordert; die Messenger-Nachricht ist unterwegs.
  • pending_approval — der Auftrag wartet auf deine Entscheidung (Badge „Genehmigung ausstehend").

Dein Agent übersetzt diese Strings im Chat oft von selbst ins Deutsche — praktisch, aber merk dir die Originale: Im Execution-Log und in technischen Antworten stehen immer die englischen Strings. Wer beide Seiten lesen kann, verfolgt jeden Auftrag ohne Rätselraten.

Zwei Arten von Grenzen

Dein selbstgebauter Agent arbeitet in genau denselben Leitplanken wie jeder andere:

  • Deine Auftrags-Limits aus dem Agentowner-Kurs gelten unverändert — sie hängen am Token, nicht am Werkzeug. Ob fertige Software oder dein n8n-Bau: „maximal 5 Aufträge pro Tag, maximal 500 € pro Auftrag" wird bei jedem prepare_order automatisch geprüft.
  • Die Rate-Limits der Plattform: Der MCP-Zugang erlaubt 30 Anfragen pro Minute, dein Token insgesamt 60 pro Minute. Für einen Chat-Agenten ist das viel — ein Workflow in einer Schleife kann es trotzdem reißen. Der Demo-Schlüssel aus den Docs ist enger begrenzt (10 pro Minute).

Wenn es hakt: die zwei Fehlersignale

  • 401 — Ausweis fehlt: n8n meldet „Authorization failed - please check your credentials", die Plattform sagt „Bearer token required". Der Token fehlt, stimmt nicht oder wurde widerrufen — prüfe die Bearer-Auth-Credential oder erstelle deinem Agenten einen neuen Token.
  • 429 — zu schnell: HTTP 429 (Too Many Requests) heißt: Rate-Limit erreicht. Kein Defekt — dein Agent hat das Tempo überzogen. Kurz warten, dann läuft es weiter; in Schleifen-Workflows baust du eine Pause ein.

Jetzt du

Prüf dich selbst — aus dem Gedächtnis abrufen baut echtes Behalten auf.

Wissens-Check

Dein Agent meldet für einen Auftrag pending_approval. Was heißt das?

pending_approval heißt „Genehmigung ausstehend": Die Plattform hat deine Freigabe angefordert, und nichts passiert, bis du entscheidest. Freigegeben ist der Auftrag erst nach deinem Ja — und gescheitert oder verworfen ist hier nichts: Dieser Status heißt Warten, nicht Scheitern.

Kurs geschafft — und jetzt?

Dein Agent steht: fünf Tools, dein Token, deine Freigabe als letzte Instanz. Wenn du weiterbauen willst, sind die API-Docs der Plattform deine Referenz — dort findest du alle 18 Tools im Detail und das SDK, falls du irgendwann ohne n8n direkt entwickeln möchtest.

Primärquelle

Sieh es dir an: Die Seite MCP Integration in den Docs der Plattform dokumentiert Statuswerte, Limits und alle Tools — dein Nachschlagewerk für alles nach diesem Kurs.
